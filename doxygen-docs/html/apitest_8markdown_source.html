3 ## Configuration
4 
5 All tests read the value of access, key and url endpoint from environment variables which need
6 to be passed on the command line, e.g. `./bucket-test S3TEST_ACCESS S3TEST_SECRET S3TEST_URL`.
7 
8 Each test driver contains multiple tests whose output is in *CSV* format:
9 
10 `<test name>,<result = `1` for pass | `0` for fail>,<blank | optional error message>`
11 
12 To test locally, two scripts to run *minio* are provided in the root folder:
13 - `minio_podman_setup.sh`: downloads and runs *minio* server with *podman*
14 - `minio_run_server.sh`: runs the pre-installed *minio* server
15 
16 The scripts print and optionally store into a file the url, access and secret to use.
17 
18 Both scripts require the following arguments:
19 - *instance name*: *minio* instance name; added into `.mc/config.json`
20 - *storage path*: directory where data are stored
21 - (optional): file name where code to set environment variable is written
22 
23 
24 It is also possible to test everything using the free `play.min.io` service.
25 Credentials for `play.min.ui` should already be present in the `.mc/config.json` file
26 created by the *minio* client.
27 
28 Each test driver will first check that an S3 endpoint is available and accessible and report an error in case of failure.
29 The test drivers will create buckets and objects named `sss-api-test-` + timestamp.
