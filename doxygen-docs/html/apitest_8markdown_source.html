2 
3 ## Configuration
4 
5 All tests read the value of access, key and url endpoint from environment variables which need
6 to be passed on the command line, e.g. `./bucket-test S3TEST_ACCESS S3TEST_SECRET S3TEST_URL`.
7 
8 Each test driver contains multiple tests printing output to `stdin` in *CSV* format:
9 
10 `<test name>,<result = `1` for pass | `0` for fail>,<blank | optional error message>`
11 
12 and checks that an S3 endpoint is available and accessible and report an error in case of failure.
13 
14 The test drivers will create buckets and objects named `sss-api-test-` + timestamp.
15 
16 Tests require an S3 endpoint to be accessible.
17 
18 One option is to use the free *play.min.io* service; other options are to
19 configure a local instance of the *minio* server or use an AWS S3 account.
20 
21 When testing with an AWS S3 account extract credentials from file `.aws/credentials`
22 and use the url: `https://s3.<region e.g. us-east-1>.amazonaws.com` as the endpoint.
23 
24 The access and secret keys for *play.min.io* are stored inside the
25 `~/.mc/config.json` file configured after installing the
26 [minio client](https://min.io/docs/minio/linux/reference/minio-mc.html).
27 
28 The script `minio_podman_setup.sh` downloads configures and runs a *minio* server instance
29 inside a container using *podman*.
30 
31 Run the script after building all the applications, making sure that
32 the `s3-gen-credentials` executable is findable through the `PATH` variable.
33 
34 Usage
35 ```sh
36 ./minio_podman_setup.sh <minio alias> <data path>
37 ```
38 Run
39 ```sh
40 chmod u+x ./minio_podman_setup.sh
41 ./minio_podman_setup.sh myalias ~/tmp/minio_data
42 ```
43 
44 In cases where the *minio* server is already installed and available, you can
45 invoke the `minio_run_server.sh` instead.
46 
47 ```sh
48 chmod u+x ./minio_run_server.sh
49 ./minio_run_server.sh myalias ~/tmp/minio_data
50 ```
51 
52 Both scripts output access, secret and URL which should be stored into
53 environment variables, the optional third argument to the scripts is the name of
54 the file where environment variables are stored: use `source` <filename> to set
55 all environment variables.
56 
57 The `test` directory includes tests for the high level interface and the S3 API.
58 
59 All tests require passing the name of the environment variables storing access,
60 secret and endpoint URL information.
61 
62 The provided `.sh` scripts inside the `test` directory can run all the tests
63 at once.
