1 # S3 Toolkit {#mainpage}
2 
3 C++ client library and tools.
4 Originally developed to test *Ceph* object storage, parallel transfers between
5 Lustre and *Ceph* on high speed networks, and learn the S3 API.
6 
7 All the code is tested on Linux SuSE and Ubuntu, x86 only and MacOS (x86 and ARM).
8 
9 ## Library
10 
11 * `libs3client`: high level functions to sign and send requests and perform
12 parallel uploads/downloads.
13 
14 S3 API accessible through the `libs3client` library, see `s3-api.h` and
15 `s3-client.h` includes.
16 
17 Check out the `app` and `test` folders for usage examples.
18 
19 A small subset of the S3 actions is implemented, but any request can be sent
20 through the `S3API::Send` method and `SendS3Request` function.
21 
22 XML requests and responses can be generated and parsed using the provided
23 high-level XML parsing and generation functions;
24 see the `Parsing` module in the Doxygen-generated API documentation or look into the
25 `xml_path.h` and `test/xml-parse-test.cpp` files.
26 
27 In the interest of keeping the interface simple:
28  - additional header parameters should be passed as required to the different
29  methods instead of relying on mapped C++ types
30  - additional information returned by the various requests should be extracted
31  from headers and body through the \c S3Api::GetResponseText
32  (returns \c std::string instance) and \c S3Api::GetResponseHeaders
33  (returns \c map instance) method
34 
35 Parallel upload and download works best when reading/writing from SSDs or RAID &
36 parallel file-systems with `stripe size = N * (part size)`.
37 
38 The code is `C++17` compliant.
39 
40 When disabling S3v4 signing the library can be used as a generic HTTP client
41 library.
42 
43 Work to enable multi-level asynchronous operations is ongoing: each thread
44 will be using async I/O for sending multiple concurrent requests.
45 
46 Parallel upload/download using MPI and integration with ADIOS are being
47 experimented as well in separate repositories.

50 ## Applications
51 
52 * `s3-client`: send requests and print raw responses
53 * `s3-presign`: generate pre-signed `URL`
54 * `s3-upload`: (single file) parallel upload
55 * `s3-download`: (singe file) parallel download
56 * `s3-gen-credentials`: generate access and secret keys
57 
58 Launch without arguments to see options.

61 ## Build and install
62 
63 The `install.sh` script includes the code for checking out the code, building
64 and installing to a user specified path.
65 Just [download](https://raw.githubusercontent.com/uv-cpp/s3client/main/install.sh)
66 and run.
67 
68 
69 ```sh
70 bash ./install.sh <install path>
71 ```
72 
73 `libcurl` must be available for the code to build.
74 
75 ...or build it manually
76 
77 E.g. Building and installing a release version under `$HOME/.local`
78 (default is `/usr/local`):
79 
80 1. `git clone --recurse-submodules https://github.com/uv-cpp/s3client.git`
81 2. `mkdir -p s3client/build`
82 3. `cd s3client/build`
83 4. `cmake ../ -DCMAKE_BUILD_TYPE=Release -DCMAKE_INSTALL_PREFIX=~/.local`;
84 5. `make -j 8`
85 6. `make install`
86 
87 Include files are found under `${CMAKE_INSTALL_PREFIX}/include/s3client`.
88 The library is found under `${CMAKE_INSTALL_PREFIX}/lib` and the executables
89 under `${CMAKE_INSTALL_PREFIX}/bin`.

99 ## Test
100 
101 In order to test the tools and API you need access to an S3 storage service.
102 One option is to use the free *play.min.io* service; other options are to
103 configure a local instance of the *minio* server or use an AWS S3 account.
104 
105 When testing with an AWS S3 account extract credentials from file `.aws/credentials`
106 and use the url: `https://s3.<region e.g. us-east-1>.amazonaws.com` as the endpoint.
107 
108 The access and secret keys for *play.min.io* are stored inside the
109 `~/.mc/config.json` file configured after installing the
110 [minio client](https://min.io/docs/minio/linux/reference/minio-mc.html).
111 
112 The script `minio_podman_setup.sh` downloads configures and runs a *minio* server instance
113 inside a container using *podman*.
114 
115 Run the script after building all the applications, making sure that
116 the `s3-gen-credentials` executable is findable through the `PATH` variable.
117 
118 
119 Usage
120 ```sh
121 ./minio_podman_setup.sh <minio alias> <data path>
122 ```
123 Run
124 ```sh
125 chmod u+x ./minio_podman_setup.sh
126 ./minio_podman_setup.sh myalias ~/tmp/minio_data
127 ```
128 
129 In cases where the *minio* server is already installed and available, you can
130 invoke the `minio_run_server.sh` instead.
131 
132 ```sh
133 chmod u+x ./minio_run_server.sh
134 ./minio_run_server.sh myalias ~/tmp/minio_data
135 ```
136 
137 Both scripts output access, secret and URL which should be stored into
138 environment variables, the optional third argument to the scripts is the name of
139 the file where environment variables are stored: use `source` <filename> to set
140 all environment variables.
141 
142 The `test` directory includes tests for the high level interface and the S3 API.
143 
144 All tests require passing the name of the environment variables storing access,
145 secret and endpoint URL information.
146 
147 The provided `.sh` scripts inside the `test` directory can run all the tests
148 at once.

150 ## Parallel data transfer
151 
152 The following considerations apply to the transfer of single large files
153 over fast connections only (at least 10 Gib/s) and to S3 services able
154 to ingest data at GiB/s rate.
155 
156 Parallel reads and writes from/to flash memory are faster than serial
157 ones, but with spinning disks the overall performance might not be
158 impacted by transferring data in parallel and even cause a slowdown.
159 
160 ### Lustre
161 
162 Lustre allows users to control the striping configuration.
163 
164 When transferring from *Lustre* make sure that the stripe
165 size is as close as possible to a multiple of the part size.
166 Use: `lfs getstripe <filename>` to retrieve the stripe size.
167 
168 When downloading data to *Lustre* make sure that the stripe size
169 for the directory is a multiple of the part size or create a file
170 with the desired stripe size first using the `lfs setstripe` command.
171 
172 ### Others
173 
174 *RAID* configurations and other parallel filesystems, like *GPFS* or *BeeGFS*
175 have preconfigured stripe/block sizes, please do refer to your system configuration
176 before configuring the number of tasks and parts per task.

282 Parse XML text using the included XML parsing framework:
283 
284 ```cpp
285  // print all objects in bucket with "last modified" date
286  XMLIStream is(req.GetContentText());
287  // when the key is not prefixed with '/' it extracts
288  // all the subtrees matching the prefix and stores
289  // them into an array of map objects
290  XMLRecords r = is["listbucketresult/contents"];
291  for (const auto &i : r) {
292  cout << "--------------------------" << endl;
293  // prefixing the key with '/' extracts the text
294  // found at the location matching the path
295  cout << i["/key"] << endl;
296  cout << i["/lastmodified"] << endl;
297  }

417 ## Parallel upload
418 
419 Parallel uploads are supported from both file and memory in the
420 API and command line application.
421 
422 Command line
423 
424 ```
425 s3-upload -a $S3_ACCESS -s $S3_SECRET -e $S3_URL -f myfile \
426  -b bucket2 -k key -j $NUM_JOBS -n $PARTS_PER_JOB -r $NUM_RETRIES
427 ```
428 
429 C++
430 
431 Extracted from the file-transfer tests.
432 
433 ```cpp
434 ...
435  #include <s3-api.h>
436 ...
437  string action = "Parallel file upload";
438  ////
439  try {
440  S3DataTransferConfig c = {.accessKey = cfg.access,
441  .secretKey = cfg.secret,
442  .bucket = bucket,
443  .key = key,
444  .file = tmp.path,
445  .endpoints = {cfg.url},
446  .jobs = NUM_JOBS,
447  .partsPerJob = CHUNKS_PER_JOB};
448  /*auto etag =*/ Upload(c);
449  S3Api s3(cfg.access, cfg.secret, cfg.url);
450  const CharArray uploaded = s3.GetObject(bucket, key);
451  if (uploaded != data)
452  throw logic_error("Data verification failed");
453  TestOutput(action, true, TEST_PREFIX);
454  } catch (const exception &e) {
455  TestOutput(action, false, TEST_PREFIX, e.what());
456  }
457 ```

460 
461 Parallel downloads are supported to both file and memory in the
462 API.
463 
464 Command line
465 
466 ```
467 s3-download -a $S3_ACCESS -s $S3_SECRET -e $S3_URL -f myfile \
468  -b bucket2 -k key -j $NUM_JOBS -n $PARTS_PER_JOB -r $NUM_RETRIES
469 ```

472 
473 Extracted from the file-transfer tests.
474 
475 ```cpp
476 ...
477  #include <s3-api.h>
478 ...
479  action = "Parallel file download";
480  try {
481  S3DataTransferConfig c = {.accessKey = cfg.access,
482  .secretKey = cfg.secret,
483  .bucket = bucket,
484  .key = key,
485  .file = tmp.path,
486  .endpoints = {cfg.url},
487  .jobs = NUM_JOBS,
488  .partsPerJob = CHUNKS_PER_JOB};
489  Download(c);
490  FILE *fi = fopen(tmp.path.c_str(), "rb");
491  vector<char> input(SIZE);
492  if (fread(input.data(), SIZE, 1, fi) != 1) {
493  throw std::runtime_error("Cannot open file for reading");
494  }
495  //
496  if (input == data) {
497  TestOutput(action, true, TEST_PREFIX);
498  } else {
499  throw logic_error("Data verification failed");
500  }
501  } catch (const exception &e) {
502  TestOutput(action, false, TEST_PREFIX, e.what());
503  }
